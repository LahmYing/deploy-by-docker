网络

  • 请求数相关
    • 每个请求都有等待时间
    • 目前资源基本是 webpack 构建出的 chunk，chunk 有助于浏览器缓存，无需针对chunk 数量即资源数做什么
    • Chrome 最多并发 6 个 TCP 连接，一个 http1 请求占一个 TCP 连接，多个 http2 请求可并发于一个 TCP 连接
  • 静态资源使用 CDN
  • http2 优势
    • 可并发于一个 TCP 连接
    • header 压缩
    • 可设置各请求优先级
  • 浏览器缓存策略
    • 304: 使用服务器缓存
    • from cache: 使用浏览器本地缓存
      • max-age 决定浏览器是否使用本地缓存而不向服务器发起请求，浏览器遇到相同文件名的资源(比如 main.js)，此时还没到 max-age，那便会使用缓存。当前基本都是经过 webpack 构建，文件名带有内容对应的hash，比如 main.${hash}.js，这样就确保了资源的唯一性

解决卡顿

  • 对于 60Hz 来说，1 帧的时长约 (1000/60)ms，除去系统上下文切换开销，每一帧中只留给我们 10ms 左右的脚本处理时间
  • js 脚本修改 DOM 造成 layout repaint，若脚本执行耗时长，这一帧耗时便长，屏幕刷新时这一帧还没更新就卡顿了，如何解决：
    • 拆分该脚本，分在几帧里执行。帧耗时缩短，帧数变多，帧频提升
    • 简化 DOM 结构
